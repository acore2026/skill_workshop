**1. Introduction**

**This paper** proposes solutions that address the following bullet
numbers within **KI#18: AI for 6G architecture and KI#19: Network for
AI**:

-   **Bullet 1: Enable the 6G CN to leverage AI capabilities (e.g., AI
    agent)**

    -   **This solution** proposes a **Skill-Based Agentic
        Architecture** that transforms the 6G CN into a dynamic \"Skill
        Registry\" where AI agents can discover and register
        capabilities.

    -   It introduces the **Agentic AI Host Function (AAIHF)** as a
        logical reasoning entity within the 3GPP SBA.

-   **Bullet 1a: Fulfilling requests when intent is included**

    -   The solution establishes a **Three-Stage Execution Pipeline**
        (Intent -\>Skill -\>Service Directive) designed specifically to
        resolve ambiguous intents into deterministic network tasks.

    -   The solution introduces a **Unified Agentic Skill Profile** to
        provide the necessary constraints and expression of intents
        required for interoperability.

-   **Bullet 1b: Fulfilling requests when intent is not included**

    -   The proposed architecture allows AI-enabled NFs to address
        standard UE requests agnostic to whether the network uses AI
        capable entities, mapping standard signaling to skills where
        applicable.

-   **Bullet 1c: Modularisation of procedures for dynamic composition**

    -   The solution defines **\"Agentic Skills\"** as modular,
        high-level capability profiles (analogous to \"Tool
        Definitions\" in Generative AI terms) that decouple the service
        logic from the physical topology.

    -   The solution enables **Skill-Based Routing**, allowing agents to
        discover any node capable of a specific task (e.g., \"Video
        Optimization\") rather than a specific node type.

-   **Bullet 3: Enable entities in 6G CN to access network AI
    capabilities**

    -   The proposed **Semantic Skill Discovery** mechanism allows
        different domains including device domain (e.g. AI-robots,
        Agentic UE), network domain (e.g. network functions), and
        application domain (e.g. cloud XR server) to register and expose
        their AI/ML capabilities (e.g., \"Generative AI-Model
        Inference\") as discoverable skills.

    -   The skills are stored in a repository, which can be a new ACRF
        (**Agentic Capability Repository Function**) or enhanced NRF.

-   **Bullet 4: Access trusted external capabilities provided by AF**

    -   **The proposed mechanism supports** Northbound Agentic Skill
        Exposure, which can explicitly enable external **Application
        Functions (AFs)** to register and expose \"Vertical Skills\"
        (e.g., Vehicle Control) via the NEF to the internal 6G CN AI
        capable entities.

Current 3GPP discovery frameworks are designed for static Network
Functions and rely on rigid, identity-based routing via exact-string
matching, which creates a \"Signaling Gap\" where any change in a
node\'s IP address or identity breaks the communication link. To
leverage the full potential of AI, the 6G CN must decouple service logic
from physical topology. By transforming the 6G CN into a dynamic
\"**Skill Registry**,\" the network enables AI Agents to operate on
declarative, high-level intents and route requests to the most suitable
\"Skill\" based on executable capabilities---such as \"Video
Optimization\"---rather than relying on a static physical address or
specific node identity. Figure 0 highlights the essential shift from
traditional identity-based networking to a new **capability-based
paradigm**.

## 6.X Solutions to KI#X

### 6.X.Y Solution #X.Y: Skill-Based Agentic Architecture

#### 6.X.Y.0 High-level solution Principles

-   **Skill-Based Agentic Architecture** transforms the 6G CN into a
    dynamic \"Skill Registry\" and **support the following:**

    -   **Agentic AI Host Function (AAIHF)**: A logical reasoning entity
        within the SBA responsible for generating intents.

    -   **Agentic Capability Repository Function (ACRF)**: An evolution
        of the NRF that uses a **Semantic Matching Engine** (e.g.,
        vector embeddings) to resolve discovery queries.

    -   **Unified Agentic Skill Profile**: A polymorphic Information
        Element (IE) containing:

        -   **Common Header**: Skill_ID (URI) and Entity_Type.

        -   **Domain Containers**: Optional constraints for Device
            (Battery/Mobility), Network (Load/Slice), or App
            (Latency/Bandwidth).

        -   **AgenticService-URI**: A dedicated endpoint for AI-driven
            command execution.

    -   **Agent-to-User Interface (A2UI)**: A declarative negotiation
        schema used to exchange structured decision spaces and skill
        options between a Network Agent and an application or user. It
        enables the refinement of ambiguous intents into a format
        suitable for the Three-Stage Execution Pipeline.

-   The solution transforms the 6G CN into a dynamic \"Skill Registry\".
    Key principles include:

    -   **Semantic Abstraction**: Discovery shifts from syntax-based
        (Exact String Matching of static IDs) to intent-based
        (Executable Capabilities).

    -   **Polymorphism**: A **Unified Agentic Skill Profile** allows
        heterogeneous entities (UE, NF, AF) to expose capabilities via a
        common header.

    -   **Decoupling**: Service logic is separated from physical
        topology, enabling routing based on \"what is needed\" rather
        than \"who is there\".

    -   **Deterministic Resolution**: A Three-Stage Pipeline ensures
        that abstract AI intents are resolved into safe,
        standard-compliant signaling.

#### 6.X.Y.1 Description

Figure X1 illustrates the fundamental architectural shift from
identity-based to skill-based networking required for the 6G system.
Traditional 3GPP discovery frameworks are built for static Network
Functions (e.g., \"SMF\") and rely on identity-based routing using
exact-string matching; this creates a \"Signaling Gap\" because any
change in a node\'s IP address breaks the communication link. The
proposed Skill-Based Agentic Architecture resolves this by decoupling
service logic from physical topology. By transforming the 6G CN into a
dynamic \"Skill Registry,\" the network enables AI Agents to operate on
declarative, high-level intents (e.g., \"Optimize Latency\") and route
requests to the most suitable \"Skill\" rather than a static physical
address. This evolution allows diverse nodes across the Device, Network,
and Application domains to be discovered based on their executable
capabilities.

**Figure X1: Transforming the 6G CN into a Serverless API Platform**

Current 3GPP discovery frameworks are designed for static Network
Functions (e.g., \"SMF\") and rely on "Identity based routing" based on
exact-string matching. If a Node changes IP, the link breaks. This
creates a \"Signaling Gap\" for 6G AI Agents that operate on high-level
intents (e.g., \"Optimize Latency\") expressed as declarative
requirements. To leverage AI, the 6G CN much decouple service logic from
physical topology.

This paper proposes a **Skill-Based Agentic Architecture**, as shown in
Figure X2, that transforms the 6G CN into a dynamic \"**Skill
Registry**\" where nodes (UE, NF, AF) are discovered based on what they
can do (Skills) rather than who they are (Identity). The network route
to the best "Skill", not a static address.

![](media/image3.png){width="3.154166666666667in"
height="2.9611111111111112in"}

**Figure X2: Skill-Based Agentic Architure**

To support natural language interaction, the architecture requires
support of the **Agent-to-User Interface (A2UI)**. The A2UI functions as
a declarative broker that exchanges structured \"Skill Options\" between
the 6G system and the user or application. This ensures that the
high-level \"Why\" (Intent) is properly negotiated and refined before
entering the deterministic signaling stages of the network. By
standardizing the A2UI, the 6G Core can support multi-vendor
interoperability where an AI agent from one provider can interpret and
act upon intents generated by user equipment or an application from
another provider.

This architecture enables a standard 3GPP network node (such as a UE or
an NF or AF/AS) to expose its functional capabilities as dynamically
discoverable **\'Agentic Skills\'**. By allowing these Skills to be
invoked via standardized **\'Service Directives\'**, effectively
transforms the telecommunications network into a modular **library of
executable functions** accessible to Generative AI.

-   **Agentic Skill:** A high-level capability profile registered within
    the network repository (NRF/ACRF), used for semantic discovery
    (e.g., mcp://skill/transport-optimization).

-   **Service Directive:** The specific function or command contained
    within a Skill (e.g., directive=\"update_qos\") that is executed by
    the target node.

**Three-Stage Execution Pipeline:** The system transforms abstract
intent into deterministic action through a strict hierarchy, ensuring
telecommunications-grade reliability while adopting AI industry
standards:

-   **Stage 1: Intent (The \"Why\")**: This is the abstract,
    non-deterministic goal generated by an application or AI Agent, such
    as natural language prompts like \"Fix the latency for the VIP
    user\".

-   **Stage 2: Skill (The \"Interface\")**: Acts as the **Discovery
    Key** or \"Tool Name,\" represented by a standardized URI (e.g.,
    4). This semantic layer allows agents to
    discover capabilities across the network without needing to know
    specific function names.

-   **Stage 3: Service Directive (The \"How\")**: The deterministic,
    executable function or 3GPP Service Operation (e.g.,
    Nsmf_PDUSession_UpdateSMContext). The target node translates the
    abstract Skill into this concrete signaling to ensure only valid,
    standard-compliant messages modify the core.

#### 6.X.2.1.1 Unified Agentic Skill Profile Structure

An Agentic Skill registered within the network (e.g., in the ACRF/NRF)
functions as a Tool Definition for connected AI agents. It can be
represented by the Agentic Skill URI (e.g.,
mcp://skill/traffic-steering) serves as the Tool Name, allowing an
LLM-based agent to identify available capabilities.

The associated Unified Agentic Skill Profile serves as the Tool Schema,
defining the expected input parameters (e.g., Target_Network_Slice,
Service_Attributes) and Policy Constraints (e.g., latency limits,
authorization levels). An Unified Agentic Skill Profile includes the
following specific attributes:

-   agenticSkillSet: A list of supported Skill IDs, represented as
    Agentic Skill URIs (e.g., \["mcp://skill/traffic-routing-opt"\] or
    \["mcp://skill/qos-assurance"\]).

-   AgenticService-URI: A specific URI dedicated to handling Agentic
    Interactions (e.g.,
    <https://aiif-01.operator.com/agentic-skill/v1/invoke>). Unlike the
    standard Service-URI (which handles legacy SBA traffic), the
    AgenticService-URI explicitly points to the endpoint capable of
    parsing the Service Directives parameter and executing the Semantic
    Translation logic. This enables a single node to support both
    standard traffic and AI-driven commands simultaneously.

The profile is a polymorphic Information Element which has A common
header defines the skill used for describing logical service
capabilities across heterogeneous domains while context-specific
containers (Device, Network Function, App) carry the domain-specific
constraints.

#### 6.X.2.1.2 Agentic Capability Repository Function (ACRF) or Enhanced NRF

The ACRF is defined herein as a logical entity responsible for the
semantic indexing and discovery of network capabilities. In various
embodiments, the ACRF functionality may be:

-   Centralized: Implemented as a standalone Network Function or a
    software module within the Network Repository Function (NRF).

-   Distributed: Implemented as a distributed logic layer where the
    Service Communication Proxy (SCP) performs the initial semantic
    interception and query routing, while the NRF maintains the backend
    index.

-   Federated: Implemented across a hierarchy of repositories, where a
    \"Global ACRF\" indexes high-level domains (e.g., \"Transport
    Domain\") and \"Local ACRFs\" index specific node capabilities
    (e.g., \"Edge Gateway Skills\").

    -   In a preferred embodiment, the ACRF utilizes a Semantic Matching
        Engine to resolve discovery queries. Unlike legacy NRFs which
        rely on rigid exact-string matching (e.g., matching \'smf-01\'
        to \'smf-01\'), the Semantic Matching Engine employs semantic
        inference logic, such as vector embedding logic or probabilistic
        mapping.

Upon receiving a discovery request containing an abstract intent (e.g.,
required-skill=\'ensure-resiliency\'), the engine converts this intent
into a high-dimensional vector representation. It then calculates the
Inference Logic between this query and the embeddings of registered
Skill IDs (e.g., mcp://skill/reliability/path-diversity). If the
semantic similarity score exceeds a pre-defined Confidence Threshold
(e.g., 0.85), the ACRF identifies a match and returns the corresponding
AgenticService-URI, enabling the network to support natural language
intents without hard-coded syntax.

-   Semantic Abstraction: Shifts the discovery mechanism from
    Syntax-Based (finding a node by its static name or IP) to
    Intent-Based (finding a node by its executable capability). This
    allows the Control Plane to understand \"What is needed\" rather
    than just \"Who is there.\"

-   Polymorphic Discovery: Enables a single discovery interface to
    resolve queries across heterogeneous domains. A single
    agenticSkillSet query can return a User Equipment (UE), a Network
    Function (NF), or an Application Function (AF), depending on which
    entity best satisfies the logical constraints (e.g., Latency vs.
    Battery).

-   Deterministic Resolution: Establishes a strict Input/Output contract
    for AI control. By defining the inputSchema (Contract) and
    dynamicState (Context) prior to execution, the system ensures that
    abstract AI \"prompts\" are converted into deterministic, safe 3GPP
    signaling messages.

-   Decoupling: Separates the Service Logic (The Task) from the Physical
    Topology (The Node). The Core AI Agent does not need to know the
    network map; it only needs to know the Skill URI.

The ACRF (or enhanced NRF) utilizes a Semantic Matching Engine to
resolve discovery queries. Unlike legacy rigid matching, the engine
employs semantic inference logic, such as vector embeddings, to
calculate similarity between an abstract intent and registered Skill
IDs.

To enable the \"Skill-Based\" operation, the Semantic Matching Engine
utilizes a semantic mapping layer (implemented within Core Agent) that
decouples the Intent from the Execution. The following table illustrates
how the single \"Skill ID\" defined in the Profile translates into

#### 6.X.2.3 Semantic Skill Discovery via ACRF/Enhanced NRF

\"Skill-Based Routing\" is enabled within the 5G/6G Control Plane. It
introduces a new Agentic Capability Repository Function (ACRF) or
extends the standard NRF (Network Repository Function) to support a new
discovery mechanism defined by the following Input/Output relationship:

-   Input Attribute (agenticSkillSet): A novel attribute within the NF
    Profile used during registration. Unlike legacy discovery which
    relies on static types (e.g., \"SMF\"), this input allows the Core
    AI Agent to search for specific logical capabilities (defined as
    Agentic Skills) regardless of the hosting node. This is essential
    for dynamically resolving the correct Interworking Gateway to
    mediate control for external domains or non-3GPP systems.

-   Output Resolution (AgenticService-URI): The specific API endpoint
    returned by the repository upon a successful semantic match. This
    URI provides the addressable Service Based Interface (SBI) target,
    enabling the Core AI Agent to immediately invoke the discovered
    skill (e.g., https://NF-01.operator.com/mcp/invoke) without prior
    knowledge of the underlying network topology.

#### 6.X.Y.2 Procedures

Editor\'s note: This clause will describe the high-level procedures and
information flows for the solution.

#### 6.X.Y.2.1 Skill-Based Discovery Procedures

This procedure shows how the ACRF/Enhanced NRF is used to register nodes
based on capability rather than identity for Sematic Discovery.

![](media/image6.emf)

**Figure 6.X.Y.2.1-1: Skill-Based Discovery Procedure**

-   Step 1, NF -\> ACRF/NRF: NF sends message to ACRF/NRF. The message
    includes Nacrf_SkillManagement_Register
    Request/Nnrf_NFManagement_NFRegister Request \| (Node_Type=\'NF\',
    agenticSkillSet).

-   Step 2, ACRF/NRF -\> NF: ACRF/NRF sends message to NF. The message
    includes Nacrf_SkillManagement_Register
    Response/Nnrf_NFManagement_NFRegister Response \| (200 OK).

-   Step 3, AAIHF (Core AI Agent): AAIHF (Core AI Agent) perform
    internal Intent ­­­­­ID(s).

-   Step 4, AAIHF (Core AI Agent) -\> ACRF/NRF: AAIHF (Core AI Agent)
    sends message to ACRF/NRF. The message includes
    Nacrf_SkillDiscovery_Request/Nnrf_NFDiscovery_Request \|
    (required-skill).

-   Step 5, ACRF/NRF -\> AAIHF (Core AI Agent): ACRF/NRF sends message
    to AAIHF (Core AI Agent). The message includes
    Nacrf_SkillDiscovery_Response/Nnrf_NFDiscovery_Response \| (List of
    Agentic_Target_Profile(s)).

#### 6.X.Y.2.2 Northbound Skill Exposure Procedure

#### This procedure enables Apps to discover network skills.

![](media/image7.emf)

**Figure 6.X.Y.2.2-1: Northbound Skill Exposure Procedure**

-   Step 1, AF/AS -\> NEF: AF/AS sends message to NEF. The message
    includes Nnef_AgentInteraction_Discovery Request \| (Semantic
    Intent-Filter).

-   Step 2, NEF -\> ACRF/NRF: NEF sends message to ACRF/NRF. The message
    includes Nacrf_SkillDiscovery_Request/Nnrf_NFDiscovery_Request \|
    (required-skill).

-   Step 3, ACRF/NRF -\> NEF: ACRF/NRF sends message to NEF. The message
    includes Nacrf_SkillDiscovery_Response/Nnrf_NFDiscovery_Response \|
    (List of Agentic Candidate Profile(s)).

-   Step 4, NEF: NEF performs Authorization Filtering \| (Authorization
    criteria -\> Allow/Disallowed skill) and Dynamic Stateful Session
    Binding.

-   Step 5, NEF -\> AF/AS: NEF sends message to AF/AS. The message
    includes Nnef_AgentInteraction_Discovery Response \| (authorized
    AgenticService URI(s)).

#### 6.X.Y.3 Impacts on Services, Entities and Interfaces

Editor\'s note: This clause captures impacts on existing services,
entities and interfaces.

-   **UE:**

    -   Support the UE_Agentic_Capability_Container to carry the Skill
        Profile from the UE to the Core Network.

-   **AMF:**

    -   Flow must be updated to handle the new capability container in
        the NAS Registration Request and route it to the ACRF/NRF.

-   **Enhanced NRF / ACRF (Agentic Capability Repository Function):**

    -   Semantic Matching Engine: The repository must evolve from rigid
        exact-string matching to semantic inference (e.g., vector
        embeddings) to resolve abstract intents into specific endpoints.

    -   agenticSkillSet Attribute: The NRF (or a new standalone ACRF)
        must support the indexing of logical \"Skills\" instead of just
        static NF types.

-   **NEF (Network Exposure Function):**

    -   Semantic Brokerage: The NEF acts as a translator between
        external \"Abstract Intents\" from Application Functions (AFs)
        and internal \"Agentic Skills\".

    -   Topology Hiding: The NEF must perform \"Capability
        Abstraction,\" masking internal infrastructure complexities
        (like specific NF IDs) while exposing sanitized, authorized
        skill endpoints to external tenants.

-   **A2UI Support \[x1\]**: The system must support a standardized
    negotiation schema (e.g., JSON-based) to carry intent-based decision
    spaces across the N33 (Northbound) and NAS (UE-to-CN) interfaces.
